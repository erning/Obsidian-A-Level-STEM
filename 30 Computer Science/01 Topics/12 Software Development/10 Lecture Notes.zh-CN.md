# Software Development 中文讲义

本讲义对应 9618 AS 大纲的 **12 Software Development（软件开发）** 部分，在 **Paper 2** 中考查。重点是从问题到可维护程序的完整旅程：该采用哪种生命周期、如何用结构图和状态转移图做设计、如何发现并分类错误，以及如何选择能拿分的测试数据和维护类型。

## 来源范围

- 大纲 12.1 Program Development Life cycle —— 分析、设计、编码、测试、维护；瀑布模型、迭代模型、RAD。
- 大纲 12.2 Program Design —— 结构图、参数、由结构图推导伪代码、状态转移图。
- 大纲 12.3 Program Testing and Maintenance —— 错误类型、测试方法、测试策略/测试计划、测试数据、维护类型。

## 1. 软件开发生命周期

**软件开发生命周期**是把问题转化成可运行、可维护程序的一组有序阶段。其意义在于可控：每个阶段都有明确目的和产出物，便于检查进度、尽早暴露问题，而不是拖到发布之后才被发现。

- **分析** —— 研究问题，收集需求，明确程序必须做什么。产出：需求规格说明。
- **设计** —— 决定如何满足需求：数据结构、模块、算法、接口、文件格式。产出：设计文档（结构图、伪代码、各种图）。
- **编码** —— 把设计翻译成某种编程语言。产出：源代码。
- **测试** —— 用计划好的数据运行代码以暴露缺陷，再定位并改正。产出：经过测试的代码和测试记录。
- **维护** —— 发布后让程序持续可用：修复缺陷、适应变化、改进性能。产出：更新后的版本。

> 阶段并不总是严格依次进行。在迭代模型和 RAD 中，团队会随着理解的深入反复回到分析和设计，但每个阶段依然会发生。

## 2. 生命周期模型对比

不同项目适合不同的周期。为稳定业务做的工资系统和需求每周都在变的初创产品原型，构建方式并不相同。

| 模型 | 原则 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|----------|
| **瀑布模型**（Waterfall） | 线性、自顶向下：完成并签字确认一个阶段后再进入下一阶段；前期文档量大 | 易于管理；里程碑清晰；文档完整；适合需求固定且充分理解的项目 | 反馈滞后——接近尾声才有可运行代码；难以回头；不适合需求多变或不明朗的项目 | 稳定、充分理解的问题；安全关键系统；需要正式合同的大型团队 |
| **迭代模型**（Iterative） | 分多轮（迭代）构建；每轮都经历分析—设计—编码—测试，产出可运行但尚不完整的版本，逐步成长 | 较早出现可运行软件；更早发现缺陷；可在迭代之间细化需求；逐步降低风险 | 管理更复杂；文档和集成工作反复出现；若不加控制容易范围蔓延 | 大型项目、需求演进中、能用早期部分成果降低风险 |
| **RAD**（快速应用开发） | 强调速度和用户反馈；使用原型、可复用组件和 CASE 工具；周期很短，用户深度参与 | 快速得到可用成果；用户能直接看到并对原型反馈；适合由可复用模块搭建的系统 | 需要高水平开发者和深度配合的用户；时间压力下质量可能下降；难用于极大或极关键的项目 | 小到中型项目、截止时间紧、需求不确定、由组件搭建的 GUI/数据库应用 |

> 识别线索：题目说“需求尚未完全明确”，考虑迭代或 RAD；说“已充分理解、固定、受监管”，考虑瀑布。

## 3. 结构图

**结构图**把问题分解为模块（子任务），并展示它们之间如何相互调用、传递什么数据。它表达的是程序的**设计**——其层次和接口——而不写代码。目的是让结构清晰可读，支持逐步求精，并让每个模块都能单独编码和测试。

### 3.1 符号

- **模块框** —— 一个命名的子任务（过程或函数）。最顶端的框是主程序，下面的框是它调用的模块。
- **顺序/调用连接符** —— 从父模块到子模块的连线，表示父模块调用子模块。
- **数据标志**（data flag） —— 箭头头部为空心，标注数据名，表示在模块之间流动的值。
- **控制标志**（control flag） —— 箭头头部为实心，标注标志名，表示 `EOF`、`Error` 之类的控制信号。

按惯例，连接线左侧的标志表示数据传入子模块，右侧的标志表示数据返回上层。

### 3.2 示例 —— 一个简单的统计程序

问题：读入一列考试成绩，然后输出平均分以及高于平均分的成绩数量。

```
                     +-------------------+
                     |   StatsProgram    |
                     +-------------------+
                       /              \
                      /                \
          +----------------+      +------------------+
          |   InputScores  |      |   ProcessScores  |
          +----------------+      +------------------+
                                         |    \
                                  Count  |     \  Average
                                         v      v
                                 +-----------------+
                                 |   CountAbove    |
                                 +-----------------+
```

参数标志说明：

- `InputScores` 向主程序返回 `Scores[]` 和 `Count`（数据标志）。
- `ProcessScores` 从主程序接收 `Scores[]` 和 `Count`，并计算出 `Average`（数据标志）。
- `CountAbove` 接收 `Scores[]`、`Count` 和 `Average`，返回 `AboveCount`（数据标志）。

### 3.3 由结构图推导伪代码

自顶向下、从左到右读图；每个模块变成一个 `PROCEDURE` 或 `FUNCTION`，各标志变成它的参数。

```pseudocode
PROCEDURE InputScores(BYREF Scores : ARRAY OF REAL, BYREF Count : INTEGER)
    OUTPUT "Enter number of scores: "
    INPUT Count
    FOR i ← 1 TO Count
        OUTPUT "Enter score: "
        INPUT Scores[i]
    NEXT i
ENDPROCEDURE

FUNCTION CalcAverage(Scores : ARRAY OF REAL, Count : INTEGER) RETURNS REAL
    DECLARE Total : REAL
    Total ← 0
    FOR i ← 1 TO Count
        Total ← Total + Scores[i]
    NEXT i
    RETURN Total / Count
ENDFUNCTION

FUNCTION CountAbove(Scores : ARRAY OF REAL, Count : INTEGER, Average : REAL) RETURNS INTEGER
    DECLARE Above : INTEGER
    Above ← 0
    FOR i ← 1 TO Count
        IF Scores[i] > Average THEN
            Above ← Above + 1
        ENDIF
    NEXT i
    RETURN Above
ENDPROCEDURE

// 主程序
DECLARE Scores : ARRAY[1..100] OF REAL
DECLARE Count, Above : INTEGER
DECLARE Average : REAL

CALL InputScores(Scores, Count)
Average ← CalcAverage(Scores, Count)
Above ← CountAbove(Scores, Count, Average)
OUTPUT "Average = ", Average
OUTPUT "Scores above average = ", Above
```

> 图上每个标志对应头部一个参数。若图上显示某个值会被修改并返回，就用 `BYREF`；如果只是读取，就按值传递。

## 4. 状态转移图

**状态转移图**通过展示算法（或某个组件）的所有可能**状态**、状态之间的**转移**，以及触发每次转移的**事件**，来记录算法。它非常适合描述任何随输入序列作出反应的事物：过程控制器、菜单、行为取决于历史记录的对象。

### 4.1 示例 —— 一笔线上订单

订单可能经过的状态：`New`、`Paid`、`Shipped`、`Delivered`。

```
   +-----+  pay     +-----+  ship    +--------+  deliver  +-----------+
   | New | -------> | Paid| ------> | Shipped | ---------> | Delivered |
   +-----+          +-----+          +--------+             +-----------+
                        |                                           ^
                        | cancel                                    |
                        v                                           |
                   +---------+   refund                             |
                   | Cancelled|  ------ (returns funds, ends) ------+
                   +---------+
```

转移：`New --pay--> Paid`、`Paid --ship--> Shipped`、`Shipped --deliver--> Delivered`、`Paid --cancel--> Cancelled`、`Cancelled --refund--> Delivered`（终结的财务结算）。读这张图，对任意当前状态和事件，你都能说出下一状态——这正是基于状态的算法所需要的。

## 5. 错误类型

| 类型 | 定义 | 示例 | 发现时机 |
|------|------|------|----------|
| **语法错误** | 违反语言语法规则的代码 | `IF x > 5 OUTPUT "hi"`（漏掉 `THEN`）；拼错的保留字 `WHIEL` | 在程序运行前的**编译/翻译阶段** |
| **逻辑错误** | 程序能运行并产出结果，但结果错误 | `Average ← Total / (Count + 1)`，应为 `Total / Count`；比较方向写反，应为 `<` 却写成 `>` | 在**运行时**，通常靠检查输出或走查发现；系统未必会报错 |
| **运行时错误** | 程序在执行中尝试了非法操作 | 除以零；访问越界下标；打开不存在的文件 | 在**运行时**，执行到出错语句时——程序崩溃或抛出异常 |

> 改法：先改语法错误（翻译器会报告）；对逻辑错误和运行时错误，先用追踪数据复现，再修改出错行并重新测试。

## 6. 测试方法

| 方法 | 做什么 | 何时使用 |
|------|--------|----------|
| **干跑**（dry run） | 程序员**在纸上**逐行追踪代码，用追踪表记录变量取值，不实际运行 | 早期检查逻辑；核对小程序段；考试题 |
| **走查**（walkthrough） | 程序员或团队**一起**逐段过代码或设计，讲解各部分作用并相互质询 | 编码/提交前评审设计或模块 |
| **白盒测试**（white-box） | 基于**代码内部结构**设计测试，覆盖每条语句、每个分支和路径 | 源代码和完整结构可获得时；对已知模块做单元测试 |
| **黑盒测试**（black-box） | 只依据**规格说明/输入和预期输出**测试，不了解内部代码 | 针对需求做功能测试；验收式检查 |
| **集成测试**（integration） | 把多个模块**组合在一起**测试，检查接口和模块间传递的数据 | 各模块单独通过后、系统测试之前 |
| **alpha 测试** | 在**开发组织内部**测试，通常由开发者或内部测试团队用真实或近似真实的数据进行 | 内部早期试运行，发布给用户之前 |
| **beta 测试** | 由**精选的一批真实最终用户**在组织外部、在自己的环境中试用 | 正式发布前夕，用以捕捉真实使用才会暴露的问题 |
| **验收测试**（acceptance） | 由**客户/最终用户**测试，判断系统是否满足约定需求并可以验收（签字确认） | 开发末期、系统上线之前 |
| **桩测试**（stub） | 编写一个**占位用的虚拟模块**，接口简单固定，使得调用它的模块能在真实子模块尚未实现时先被测试 | 自顶向下开发和测试，被调用模块尚未编写时 |

> 白盒是往盒子**里面**看（结构）；黑盒把盒子当作不透明（规格）。alpha 在**内部**；beta 是**外部用户**；验收是**客户签字**。

## 7. 测试策略、测试计划与测试数据

**测试策略**是总体方针：测试的范围、采用的方法、谁测什么，以及测试顺序（单元 → 集成 → 系统 → 验收）。它回答的是"我们如何确保这个系统可以发布？"。

**测试计划**是由策略派生出来的具体测试清单，每条通常包含：

- 测试编号和描述；
- 采用的方法（干跑、白盒等）；
- 使用的**测试数据**；
- **预期结果**；
- 记录**实际结果**和通过/失败判定的位置。

### 7.1 选择测试数据

| 类别 | 含义 | 对一个 `0–120` 的年龄字段 |
|------|------|--------------------------|
| **正常数据** | 典型、合法、程序应当处理的值 | `25`、`40`、`67` |
| **异常数据**（错误数据） | 类型错误或明显非法，程序应当拒绝 | `"twenty"`、`-5`、`200`、空输入 |
| **极端数据** | 合法范围最边缘的值 | `0` 和 `120` |
| **边界数据** | 边界两侧及边界上的值，用于检查比较运算符 | `0`、`1`、`119`、`120`，以及刚出界的 `-1`、`121` |

> 极端数据测试合法范围的端点本身；边界数据围绕边界测试，用于捕捉差一错误（`<` 写成 `<=`）。两者都要给，另外至少给一个异常用例。

## 8. 维护类型

程序发布后需要**维护**。大纲区分了三种修改原因。

| 类型 | 定义 | 示例 |
|------|------|------|
| **完善性维护** | 超出原规格地改进系统——新功能、更优性能、更好的易用性 | 增加"打印为 PDF"选项；让报表跑得更快 |
| **适应性维护** | 改动系统，使其能在**新环境**中继续运行 | 迁移到新操作系统、新的数据库版本，或政府更新的新税则 |
| **纠正性维护** | **修复发布前未发现的缺陷** | 修正一个算错增值税的公式；修复输入为空时的崩溃 |

> 完善性 = "让它更好用"；适应性 = "世界变了，让它继续能用"；纠正性 = "修掉一直存在的错"。

## 做题顺序

1. 通读情景，判断需求是固定的还是变化的——由此选定生命周期模型。
2. 设计题先列出输入和输出，再把工作拆成各负其职的模块。
3. 画结构图：主程序在顶，子模块在下，给每个数据标志和控制标志都加上标签。
4. 自顶向下读图写伪代码；把每个标志变成一个参数，只有当值必须改变时才用 `BYREF`。
5. 测试题先指出方法，再分类别给数据（正常、异常、极端/边界），并写出预期结果。
6. 维护题问自己：是新功能、新环境、还是修缺陷——然后分别标为完善性、适应性或纠正性。

## 常见错误

- 对需求一直在变的项目选择瀑布模型。
- 在结构图上漏掉数据标志/控制标志，或画了箭头却不加标签。
- 把极端数据（端点本身）和边界数据（刚内刚外）搞混。
- 混淆 alpha（内部）和 beta（外部用户），或混淆 beta 与验收（客户签字）。
- 把修缺陷称作"适应性维护"——适应性针对的是**新环境**，不是纠正旧缺陷。
- 只列正常测试数据，漏掉异常和边界用例。
- 把桩说成是完成好的模块——桩只是测试用的**占位**虚拟模块。

## 快速自查

- 说出生命周期的五个阶段，并各给一个产出物。
- 对"某个初创团队在开发功能每周都在变的应用"，说明为什么选迭代或 RAD 而非瀑布。
- 为一个工资程序画结构图：输入工时、计算工资、打印工资单，并标出参数标志。
- 从该结构图写出 `CalculatePay` 模块的伪代码。
- 为一个必须为 8–20 个字符的密码，给出正常、异常、极端和边界数据。
- 分类判断：增加深色模式；迁移到新数据库；修复一个除以零的崩溃。
