# Software Development Review Checklist

Use this as a self-test after reading the notes and attempting the practice problems. Keep answers out of sight until you can recall, trace, and explain the ideas without prompting.

## Ideas to Recall

- [ ] Recall the five life-cycle stages: analysis, design, coding, testing, and maintenance.
- [ ] State one deliverable from each life-cycle stage.
- [ ] Distinguish waterfall, iterative, and RAD by principles, benefits, drawbacks, and project fit.
- [ ] Explain the purpose of a structure chart.
- [ ] Recall structure-chart symbols: module boxes, call connectors, data flags, and control flags.
- [ ] Explain how parameters in pseudocode come from data and control flags.
- [ ] Define a state-transition diagram in terms of states, events, and transitions.
- [ ] Distinguish syntax, logic, and run-time errors by when they are detected.
- [ ] Recall dry run, walkthrough, white-box, black-box, integration, alpha, beta, acceptance, and stub testing.
- [ ] Distinguish a test strategy from a test plan.
- [ ] Distinguish normal, abnormal, extreme, and boundary test data.
- [ ] Distinguish perfective, adaptive, and corrective maintenance.

## Skills to Perform

- [ ] Choose a life-cycle model from a scenario and justify it against a less suitable model.
- [ ] Outline what happens in each life-cycle stage for a small project.
- [ ] Draw a structure chart with one-job modules and labelled flags.
- [ ] Read a structure chart top-down and derive module pseudocode from it.
- [ ] Decide which parameters should be passed by value and which by reference.
- [ ] Draw a state-transition diagram for a simple system.
- [ ] State what happens when an event has no transition from the current state.
- [ ] Classify faults as syntax, logic, or run-time with a reason.
- [ ] Choose a testing method from the testing goal and available information.
- [ ] Create test data that covers normal, abnormal, extreme, and boundary cases.
- [ ] Build a simple test-plan entry with data, expected result, actual result, and verdict.
- [ ] Classify a maintenance change and justify the classification.

## Things to Trace or Explain

- [ ] Trace how a requirement moves from analysis into design, coding, testing, and maintenance.
- [ ] Explain why changing requirements favour iterative or RAD development over a strict linear model.
- [ ] Trace parameter flow through a structure chart and into routine headers.
- [ ] Explain the difference between a data flag and a control flag.
- [ ] Trace a state machine from one state to the next for a sequence of events.
- [ ] Explain why a run-time error depends on a particular execution path or input.
- [ ] Explain why white-box testing can target branches that black-box testing may not see.
- [ ] Explain how a stub lets a calling module be tested before a called module exists.
- [ ] Trace how boundary values catch off-by-one comparison errors.
- [ ] Connect this topic to [[30 Computer Science/01 Topics/11 Programming/00 Overview|Programming]] by explaining how routines are designed, tested, and changed.

## Common Errors to Avoid

- [ ] Choosing waterfall for a project whose requirements are unclear or changing.
- [ ] Describing RAD as simply "coding quickly" without user feedback and prototyping.
- [ ] Drawing a structure chart with arrows but no labels.
- [ ] Treating a structure chart as a flowchart of execution decisions.
- [ ] Forgetting to show returned data from a child module.
- [ ] Confusing extreme data with boundary data.
- [ ] Giving only normal test data.
- [ ] Confusing alpha testing with beta testing or acceptance testing.
- [ ] Calling a bug fix adaptive maintenance.
- [ ] Treating a stub as the completed module.
- [ ] Classifying a wrong answer as a syntax error when the program can still run.

## Ready to Move On When

- [ ] I can choose a development model from the project conditions and justify the choice.
- [ ] I can draw and read structure charts with accurate parameter flow.
- [ ] I can derive routine pseudocode from a structure chart without losing parameter direction.
- [ ] I can draw and explain a state-transition diagram for a reactive process.
- [ ] I can classify errors, tests, test data, and maintenance changes from scenarios.
- [ ] I can design a compact test plan that covers valid, invalid, and edge-case data.
- [ ] I can connect this topic to [[30 Computer Science/01 Topics/09 Algorithm Design and Problem-solving/00 Overview|Algorithm Design and Problem-solving]] through decomposition and refinement.
- [ ] I can connect this topic to [[30 Computer Science/01 Topics/20 Further Programming/00 Overview|Further Programming]] through robust file processing, exception handling, and maintainable design.
