# Data Representation 中文讲义

本主题是 AS 阶段第 1 章 Information Representation 的延伸。它把问题从"单个值怎么存"扩展到"如何描述一个完整的问题、如何在磁盘上保存记录、如何在尽量少损失精度的前提下表示实数"。全部内容由 Paper 3 考查。

## 来源范围

CAIE 9618 大纲第 13 章，共三个子主题：

- **13.1** 用户自定义类型——枚举、指针、集合、记录、类/对象
- **13.2** 文件组织与存取——串行、顺序、随机；哈希
- **13.3** 浮点数——格式、转换、规范化、误差

## 1. 用户自定义类型

### 1.1 为什么需要它

内置类型（`INTEGER`、`REAL`、`BOOLEAN`、`CHAR`、`STRING`）各自只能描述一个值。真实问题里的对象往往由多个相关字段组成（一名学生有姓名、成绩、班级），或取值只能来自一个固定列表（七天、四种花色）。用户自定义类型让你把这样的结构命名一次后反复使用，代码因此更清晰、类型更安全、更易维护。

分两大类：

- **非复合类型**——由单个已有类型构造：枚举、指针。
- **复合类型**——由多个分量构造：集合、记录、类/对象。

### 1.2 枚举（非复合）

```pseudocode
TYPE Day = (Monday, Tuesday, Wednesday, Thursday, Friday, Saturday, Sunday)
```

这些值是有序的命名常量，本身没有算术含义。内部映射为整数 0、1、2……，所以比较和 `FOR` 循环都能用。

使用示例：

```pseudocode
DECLARE today : Day
today ← Wednesday
IF today = Friday THEN
   OUTPUT "Almost the weekend"
ENDIF
```

当变量只能取少数几个已知值之一时，就用枚举——它能在设计阶段挡住拼写错误和非法取值。

### 1.3 指针（非复合）

```pseudocode
TYPE IntPtr = ^INTEGER
```

指针存的是另一个变量的*地址*，而不是值本身。正因如此，才能构造出运行时动态增删的链表、树等结构。

使用示例：

```pseudocode
DECLARE p : IntPtr
DECLARE x : INTEGER
x ← 42
p ← @x          // p 保存 x 的地址
OUTPUT p^       // 输出 42（解引用）
p^ ← 99         // 把 x 改成 99
```

### 1.4 集合（复合）

```pseudocode
TYPE CharSet = SET OF CHAR
```

集合保存来自同一基类型的若干取值，元素不重复，也不分先后。典型操作是判定成员、并集、交集、差集。

使用示例——检查一个单词里出现了哪些元音：

```pseudocode
TYPE VowelSet = SET OF CHAR
DEFINE Vowels() : VowelSet
   ...
ENDDEFINE
```

当你只关心"某元素在不在里面""把几组合并"，而不在意顺序时，就用集合。

### 1.5 记录（复合）

```pseudocode
TYPE Student
   DECLARE Name : STRING
   DECLARE Mark : INTEGER
   DECLARE Grade : CHAR
ENDTYPE
```

记录把若干个名字不同、类型也可能不同的字段组合在一个名字下，正好对应文件的一行或数据库的一个元组。

使用示例：

```pseudocode
DECLARE s : Student
s.Name ← "Ada"
s.Mark ← 87
s.Grade ← 'A'
OUTPUT s.Name, s.Mark
```

### 1.6 类/对象（复合）

类是一种"还携带了方法"的记录，方法即作用在这些数据上的过程，并支持封装。对象是类的实例。当行为天然属于数据本身时，就设计一个类——详见 Further Programming。

### 1.7 针对问题选择类型

| 问题中的线索 | 适用类型 |
|---|---|
| 一组固定的命名取值 | 枚举 |
| 需要引用/链接另一个值 | 指针 |
| 一组元素、做成员判断、无顺序 | 集合 |
| 多个不同类型的命名字段 | 记录 |
| 字段**连同**作用于它们的操作 | 类/对象 |

## 2. 文件组织

*组织方式*指记录在存储介质上物理排列的方式。

| 组织方式 | 记录如何存放 | 典型用途 |
|---|---|---|
| **串行**（Serial） | 一条接一条，按到达顺序 | 备份、事务日志——成批处理 |
| **顺序**（Sequential） | 按**关键字段**排序 | 周期性批处理：发工资、月度账单 |
| **随机**（Random，直接） | 每条记录的地址由**记录键**经哈希函数算出 | 交互式查询：查余额、查库存 |

- 串行和顺序文件从头到尾顺序读很便宜，但要找某一条记录就很慢。
- 随机文件可以一步跳到目标记录，因此适合需要快速单条访问的实时系统。

## 3. 文件存取

*存取方式*指允许的读写方式。

| 文件组织 | 顺序存取 | 直接存取 |
|---|---|---|
| 串行 | ✔ | ✘ |
| 顺序 | ✔ | ✔ |
| 随机 | ✘（不自然） | ✔ |

- **顺序存取**——从头开始一条接一条地读。串行和顺序文件都可用。
- **直接存取**——按地址/偏移一步定位到记录，不必先读前面的。顺序文件（按键序中的位置算偏移）和随机文件（按哈希得到地址）都可用。

## 4. 哈希算法

哈希函数把记录键转换成文件地址，使随机文件能通过一次计算放置并随后取回一条记录，而无需查找。

**取模哈希**是最典型的例子：

```
address = key MOD numberOfRecords
```

示例——把以 `EmployeeID` 为键的记录存进一个有 11 个槽的随机文件：

| EmployeeID | 1754 | 3200 | 4429 | 5061 |
|---|---|---|---|---|
| Address = key MOD 11 | 5 | 10 | 7 | 1 |

要读 4429 号员工，算 $4429 \bmod 11 = 7$，直接跳到第 7 个槽即可，不需要扫描。

**冲突**（collision）发生在两个键算出同一地址时。1754 和 1765 都得 5，因为 $1765 \bmod 11 = 5$。冲突的处理方式有两种：用**溢出区**（把多出来的记录链接到该槽后），或者**再哈希**（往后探测下一个空槽）。

## 5. 浮点数格式

一个二进制浮点数由两部分组成，都用**二补码**存储：

- **尾数**（mantissa）——有效位，按二进制小数存储（小数点紧挨符号位之后）。
- **阶码/指数**（exponent）——乘以 2 的多少次方。

即 $\text{value} = \text{mantissa} \times 2^{\text{exponent}}$。

常见的考题规格是 12 位 = 8 位尾数 + 4 位阶码，例如 $01011100\;0011$。

### 尾数与阶码的取舍

总位数固定时，分给一边的位必然从另一边拿走。

| 调整 | 影响 |
|---|---|
| 给**阶码**更多位 | **范围**更大——能表示极大和极小的量级 |
| 给**尾数**更多位 | **精度**更高——有效位更多，舍入误差更小 |

两者无法同时最大化，大纲要求你能针对具体问题为某种分配方案给出理由。

## 6. 转换

### 6.1 二进制浮点 → 十进制

示例：8 位尾数、4 位阶码。值为 $01100000\;0011$。

1. 阶码 $0011 = +3$。
2. 尾数 $01100000$——小数点在符号位之后 $\rightarrow 0.1100000$。
3. $0.1100000_{2} = \tfrac{1}{2} + \tfrac{1}{4} = 0.75$。
4. 值 $= 0.75 \times 2^{3} = 0.75 \times 8 = \mathbf{6.0}$。

### 6.2 十进制 → 二进制浮点

用 8 位尾数、4 位阶码表示 5.75。

1. $5.75 = 101.11_{2}$。
2. 规范化成 `0.1…` 形式：$101.11 = 0.10111 \times 2^{3}$。
3. 尾数（8 位，小数点在符号位后）：$01011100$。
4. 阶码 $+3$（4 位）：$0011$。
5. 结果：$01011100\;0011$。

**负数**的规范化尾数以 $10$ 开头（如 $10100000$）；做法是先写出绝对值，再取二补码。

## 7. 规范化

**是什么：** 把尾数左移，使正数以 $01$ 开头、负数以 $10$ 开头，同时相应调整阶码。

**为什么：** 这样小数点恰好落在第一个有效位之后，每一位尾数都承载信息 $\rightarrow$ 在该字段宽度下**精度最大**，且每个值只有唯一的表示。

示例——规范化 $00011000\;0101$（8 位尾数、4 位阶码）：

1. 未规范化值：尾数 $0.0001100 = \tfrac{3}{32}$，阶码 $+5$ $\rightarrow \tfrac{3}{32} \times 32 = 3.0$。
2. 把尾数左移 3 位，直到以 $01$ 开头：$00011000 \rightarrow 01100000$。
3. 阶码减 3 以保持值不变：$5 - 3 = 2$。
4. 结果：$01100000\;0010$。
5. 验算：$0.1100000 \times 2^{2} = 0.75 \times 4 = \mathbf{3.0}$。 ✓

经验法则：尾数每左移一位，阶码就减 1。

## 8. 近似、下溢、溢出、舍入误差

有限的二进制字段不可能表示每一个实数。由此带来四类后果：

- **近似。** 许多十进制小数没有精确的二进制形式。$0.1_{10} = 0.000\overline{1100}_{2}$，所以存储 0.1 永远略有偏差——这也是大多数语言里 `0.1 + 0.2 != 0.3` 的原因。
- **舍入误差。** 当真实值需要的位数超过尾数所能容纳时，存储值被舍入到最近的可表示数；丢掉的那些位造成小误差，并可能在多次运算中累积。
- **下溢。** 结果太接近零以至于无法表示（量级小于最小规范化数）时下溢为 0。在一个最小数约为 $10^{-38}$ 的系统里给 0 加 $10^{-40}$，结果就是 0。
- **溢出。** 结果量级太大、阶码字段装不下时溢出——阶码位无法表示所需的 2 的幂，于是该数被存作"无穷大"或视为未定义。

## 做题顺序

1. **类型题**——盯住题中的名词：一*组名字*？记录。一*组固定的状态*？枚举。*链接*？指针。*字段加行为*？类。
2. **文件题**——问两件事：(a) 批处理还是交互式？(b) 有没有可用的键？批处理且有键 → 顺序；交互式且有键 → 随机。
3. **浮点转换**——先分开尾数和阶码，各自按二补码解码，再合并。
4. **规范化**——先确定符号位，把尾数移到首位与符号匹配（$01$ 或 $10$），阶码朝相反方向调整。

## 常见错误

- 使用了未在 `TYPE` 行声明过的枚举值。
- 把*组织方式*（串行/顺序/随机）与*存取方式*（顺序/直接）搞混。
- 忘了阶码是**带符号**的——$1011$ 是 $-5$，不是 $11$。
- 规范化时忘了补偿阶码——只移尾数会改变值，必须同时调整阶码。
- 误以为 0.1 在二进制里能精确表示。
- 小数点位置放错——按大纲约定，它紧挨符号位之后。
