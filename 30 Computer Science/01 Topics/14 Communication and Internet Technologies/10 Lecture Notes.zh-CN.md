# Communication and Internet Technologies 中文讲义

网络上的计算机不是把字节发出去就万事大吉，它们必须遵循约定的**协议**，这样不同厂家的机器才能彼此听懂；而数据究竟怎么从发送方走到接收方，则由**交换**方式来安排。这一节是 [[30 Computer Science/01 Topics/02 Communication/00 Overview|AS 第 2 节 Communication]] 的延续：AS 部分讲的是构建一个网络所需的硬件、拓扑和编址，这里则讨论让主机彼此对话的规则（**TCP/IP 协议族**及其应用层协议），以及数据究竟如何穿越互联网（线路交换与分组交换，以及路由器的作用）。

## 来源范围

本讲义对应 CAIE 9618 A Level 第 14 节 Communication and Internet Technologies，在 **Paper 3** 中考查：

- 14.1 Protocols：协议为何必不可少；协议栈与各层；**TCP/IP 协议族**的四个层（应用层、传输层、网际层、链路层）；主机间发送消息时应用层发生的事；应用层协议 **HTTP、FTP、POP3、IMAP、SMTP、BitTorrent**。
- 14.2 Circuit switching, packet switching：两种交换各自如何工作、优缺点和适用场景；**路由器**在分组交换中的作用；分组交换如何让消息穿越网络（含互联网）。

## 1. 协议为何必不可少；协议栈与层

协议（protocol）是两台计算机通信时必须共同遵守的一组约定——消息的格式、发送顺序、每个字段的含义以及出错时如何处理。如果没有协议，发送方和接收方就没有共同语言：字节到了也无人理解其含义，就连“下一步轮到谁发送”都无法确定。协议之所以必不可少，正是因为通信两端是彼此独立的机器，对话的每一个细节都得事先讲好。

通信所需的功能太多，无法由单一协议承担，于是被拆成一个**协议栈**，分成若干个**层**。每一层：

- 只负责一项明确的**功能**（如物理信号、寻址、表示应用数据）；
- 为**正上方**的层提供服务，并调用**正下方**的层提供的服务；
- 在逻辑上与对方机器上**同一层**的对等实体通信，尽管物理上数据是经由下面各层传过去的。

这种分层让每一层都可以独立设计和替换：应用层根本不必关心物理链路是光纤还是 WiFi，因为这些细节已被下层屏蔽。发送方的应用产生消息后，沿栈**向下**传递，每经过一层就加上本层的头部（控制信息）；接收方再沿栈**向上**传递，每层读取并剥掉自己的头部，最终原始数据到达接收方的应用。

## 2. TCP/IP 协议族与四个层

TCP/IP 协议族是支撑互联网运行的一整套分层协议，共**四个层**。每一层都有其目的（purpose，为谁服务）和功能（function，具体做什么）。

| 层 | 目的 | 功能 |
| --- | --- | --- |
| **应用层**（Application） | 给用户程序提供接口和服务（网页、邮件、文件传输） | 规定应用层消息的格式和协议（HTTP、FTP、SMTP 等），把用户要发的数据编码，交给传输层 |
| **传输层**（Transport） | 在通信的两个程序之间提供端到端的投递 | 把数据拆成**段**，用**端口号**标识应用，并处理可靠性：**TCP** 给段编号、重传丢失段，保证可靠有序；**UDP** 发数据报无保证，换取速度 |
| **网际层**（Internet） | 让数据包跨越多个网络，从源主机送到目标主机 | 加上 **IP 地址**（源和目标）构成**数据包**，由路由器逐跳**路由**跨越网络；处理与物理介质无关的逻辑寻址 |
| **链路层**（Link） | 把数据送到单个物理链路上的下一个节点 | 把数据包封装成**帧**，加上本段链路的硬件 **MAC 地址**，在铜线、光纤或无线介质上完成物理信号传输，把帧送到该链路上的下一台设备 |

可以看出各层如何分工：应用层知道**发什么**，传输层保证**送达**，网际层负责**跨网络路由**，链路层负责**一段物理链路**上的搬运。

### 消息沿协议栈下传与上传

主机间发送消息（比如浏览器请求网页）时，数据沿发送方的四层**向下**走，经过介质，再沿接收方的四层**向上**走，每一层都与对端的对等层对话：

```
   发送方主机                                 接收方主机
   ---------                                  --------
                                              ^  4. 应用层      （浏览器渲染页面）
   1. 应用层    （浏览器构造 GET）           |  3. 传输层      （TCP 重组、确认）
   2. 传输层    （TCP 加端口号、序号）        |  2. 网际层      （IP 剥去地址）
   3. 网际层    （IP 加源/目标地址）          |  1. 链路层      （网卡读取帧）
   4. 链路层    （网卡把数据送到线上）        |
        |                                     |
        +-----> 物理介质 / 网络 / 路由器 ----->+
```

发送方每一层都加上自己的头部，接收方对应的那一层则读取并解释该头部。两个应用层表现得像在直接对话，尽管物理上数据已被层层封装、送上线路、由路由器转发，最后又被重新拼好。

### 应用层发生的事

互联网上主机之间发送消息时，**应用层**是用户可见数据被创建和格式化的地方——而且是按某项具体服务来格式化。比如请求网页时，浏览器构造一条 **HTTP** 请求消息（方法、URL、头部、正文）；发邮件时，邮件客户端构造一条 **SMTP** 消息。应用层不操心地址和线缆，它按自己的协议产出格式正确的消息，交给下方的**传输层**，把送达之事托付给各层。在接收主机上，应用层拿到重组好的数据并据此行动（Web 服务器读出 HTTP 请求并返回页面）。

## 3. 应用层协议

每个应用层协议都服务于一个目的，承载一种数据。要记住每个协议做什么、承载什么。

| 协议 | 目的 | 承载内容 |
| --- | --- | --- |
| **HTTP**（超文本传输协议） | 在浏览器和 Web 服务器之间传输网页及网页资源 | 网页（HTML）、图片等以 URL 标识的资源；请求方法（GET、POST）和响应 |
| **FTP**（文件传输协议） | 在客户端和服务器之间传输文件，并支持目录列表 | 上传或下载的整个文件；使用分开的控制连接和数据连接 |
| **POP3**（邮局协议第 3 版） | 把服务器邮箱中的邮件取到单一客户端 | 下载的邮件消息；通常下载后就把邮件从服务器删除 |
| **IMAP**（互联网消息访问协议） | 让客户端访问并管理存放在服务器上的邮件 | 留在服务器上的邮件；支持多个客户端读取同一邮箱、进行文件夹管理 |
| **SMTP**（简单邮件传输协议） | 把邮件从客户端发送到邮件服务器，以及在邮件服务器之间转发 | 正在被推向目的地的外发邮件 |
| **BitTorrent** | 提供**对等（peer-to-peer）文件共享** | 被切成小块的文件数据，在大量对等节点之间直接交换（每个节点既是下载者又是上传者）；没有中央服务器存放完整文件 |

一种好记的分组方式：**HTTP/FTP** 搬运网页和文件；**POP3、IMAP、SMTP** 是邮件三件套（SMTP 负责发送，POP3 和 IMAP 负责接收——但 IMAP 把邮件留在服务器上）；**BitTorrent** 是其中的异类，因为它是**对等**的，每个参与者既上传又下载。

## 4. 线路交换

线路交换（circuit switching）在发送任何数据**之前**，先在发送方和接收方之间建立一条专用的通信**路径（电路）**，并在整个通话期间一直保持。经典例子是**传统电话网**：你打电话时，沿途的交换机端到端地预留一条连续路径，并一直保留——哪怕没人说话——直到你挂机。

整个过程分三个阶段：**呼叫建立**（预留路径）、**数据传输**（这条路径专属于你）、**呼叫拆除**（释放路径供他人使用）。

好处：

- 电路一旦建立，**数据率有保证**且恒定。
- 连接后**延迟低且可预测**，因为路径固定，没有逐包的路由决策。
- 对连续数据很合适，适合语音这类稳定的数据流。

缺点：

- 专用路径在空闲时被**浪费**——没人说话时，预留的带宽白白闲着。
- **呼叫建立需要时间**，通信无法立即开始。
- **不灵活**：路径上任何一条链路或交换机出故障，整通电话就断，必须重新建立。
- 不擅长高效处理来自众多不同来源的数据突发。

适用场景：传统电话网，以及任何在通话期间需要恒定速率信道的场合。

## 5. 分组交换

分组交换（packet switching）里**没有专用路径**。数据被切成一个个小的**数据包**，每个包都带自己的头部，含源地址和目标地址。每个包被送进网络，**独立地**被路由到目的地——它们可能走不同的路径、乱序到达，再由接收方重新拼装。这正是**互联网**的工作方式。

好处：

- **带宽利用率高**：许多用户共享同一条链路；某个源静默时别的源就能占用容量，资源不被浪费。
- **健壮**：某条链路或路由器失效时，数据包可以**改走其他路径**，没有任何单点会让整段旅程中断。
- **发送前无需建立**——包一成形即可发出。
- 适合**突发性**数据（网页请求、邮件），即流量忽高忽低的情况。

缺点：

- **延迟可变**：包在繁忙的路由器处可能排队，到达时间各不相同。
- **没有保证的数据率**——负载很重时，包可能被延迟甚至丢弃。
- **开销**：每个包都要带头部，重组和差错校验也增加处理量。
- 包可能**乱序到达**，需要接收方重新排序。

适用场景：互联网，以及任何承载众多独立、突发数据流的网络。

## 6. 路由器在分组交换中的作用；消息如何穿越网络

路由器（router）是让分组交换能跨多个网络工作的设备。它在分组交换中的功能是：

- 读取每个到达数据包的**目标 IP 地址**；
- 查询自己的**路由表**，表中记录着通往各目标网络的链路；
- 为该数据包选择**最佳的下一跳**，并在对应链路上**转发**；
- 若某链路拥塞，则丢弃或排队该数据包。

每个路由器只决定**下一跳**，而不是整条路由，因此路径可以随着链路的通断而自适应变化。

### 数据包穿越网络的过程

设想网络 1 上的主机 A 向网络 2 上的主机 B 发送一个数据包，两者相距数跳：

```
[ 主机 A ]--网络 1--[ 路由器 1 ]----[ 路由器 2 ]----[ 路由器 3 ]--网络 2--[ 主机 B ]
   源 IP = A                                            （每个路由器读取目标 IP = B，
   目标 IP = B                                            选出通往 B 的下一跳）
```

1. **A 的应用层**创建数据（比如一条 HTTP 请求），沿协议栈向下传递；**网际层**给它盖上源 IP = A、目标 IP = B，构成数据包。
2. 数据包离开 A，进入**网络 1**，到达**路由器 1**。
3. **路由器 1**读取**目标 IP 地址（B）**，在**路由表**中查到，最佳的**下一跳**是去往路由器 2，于是沿该链路转发。（路由器 1 不必知道整条路径，只需知道下一步。）
4. 数据包到达**路由器 2**，同样读取目标 IP、查路由表，转发给路由器 3。
5. **路由器 3**读取目标 IP，发现此包现在属于 B 所在的**网络 2**，于是把它投递到网络 2 上。
6. 数据包到达**主机 B**，其协议栈**向上**传递：链路层读取帧，网际层核对 IP 地址，传输层重组并回送确认，**应用层**最终收到原始数据并据此行动。

要点：每个数据包都按其目标 IP 地址被**独立**路由；路由器不为每段对话保留状态；如果某个路由器或链路失效，后续数据包可以改走别的路由器；目的地主机的协议栈最后负责把一切重组完毕。

## 7. 线路交换与分组交换对比

| 方面 | 线路交换 | 分组交换 |
| --- | --- | --- |
| **资源占用** | 整个通话期间预留专用路径；**空闲时浪费** | 多用户共享链路；**无空闲浪费**，带宽可复用 |
| **延迟** | 连接后低且恒定；数据流之前有**建立延迟** | 延迟可变；包可能排队；**无建立延迟** |
| **健壮性** | 脆弱——路径上某条链路失效，**整通电话中断** | 健壮——数据包可绕过失效的链路/路由器**改道** |
| **速率保证** | 保证恒定速率 | 无保证，速率随负载波动 |
| **计费方式** | 通常按通话**时长**（连接时间）计费 | 通常按传输的**数据量**（每包/每 Mb）计费 |
| **适用数据** | 连续、恒定速率的数据流（语音） | 突发的、彼此独立的数据流（网页、邮件、文件） |
| **典型用途** | 传统电话网 | 互联网 |

一个简单的记忆法：线路交换相当于为你的整段旅程预留一条“私家路”；分组交换则把你的数据切成一块块，扔进“公共路网”，由网络逐块路由到目的地。

## 做题顺序

1. 遇到“为什么需要协议”的题，先点明：两台独立的机器必须就消息格式、顺序和差错处理达成一致，没有协议就没有共同语言。
2. 遇到 TCP/IP 分层的题，逐层说出四个层，给出它的目的**和**一项具体功能；切勿只列层名。
3. 追踪主机间消息时，画出它沿发送方**向下**、经过介质、再沿接收方**向上**的过程，标注每层加了什么。
4. 给应用协议配对时，回想分组：网页/文件（HTTP、FTP）、邮件（SMTP 发送，POP3/IMAP 接收），以及唯一对等的 BitTorrent。
5. 遇到交换方式的题，先判断场景看重的是恒定有保证的信道（线路）还是共享、健壮、突发的投递（分组），再用表格给出理由。
6. 遇到“数据包如何穿越网络”的题，让一个包依次走过每个路由器：读目标 IP、查路由表、转发下一跳——最后以目的地重组收尾。

## 常见错误

- 只列出 TCP/IP 的四个层，却说不出每层做什么；分数在功能上，不在名字上。
- 把协议和它所在的层搞混——比如把“HTTP 就是应用层”说成正确说法，而应是“HTTP 运行在应用层”。
- 忘记 **BitTorrent 是对等的**，每个节点既上传又下载，没有中央服务器存放完整文件。
- 混淆 POP3 和 IMAP：POP3 通常下载后从服务器删除；IMAP 把邮件留在服务器上，多个客户端可共享同一邮箱。
- 把线路交换说成“没有延迟”——它有建立延迟，且预留路径空闲时被浪费。
- 说分组交换保证送达：并非如此。包可能延迟、丢失或乱序到达；可靠性和重传（如需要）由目的地处理（传输层的 TCP）。
- 说路由器知道数据包的整条路径；每个路由器只从路由表中选出**下一跳**。
- 忘记计费差异：线路交换通常按时长计费，分组交换通常按数据量计费。
- 把帧（链路层，MAC 地址）和数据包（网际层，IP 地址）搞混。
